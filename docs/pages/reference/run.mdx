# rollup.run

Convenience request loop: repeatedly [`waitForInput`](/reference/wait-for-input)s for the next request — accepting or rejecting the previous one — and dispatches it to your handlers.

## Usage

```ts twoslash
import { Rollup, decodeAdvance, encodeNotice } from '@tuler/node-libcmt';

const rollup = new Rollup();

await rollup.run({
    advance(request, rollup) {
        const advance = decodeAdvance(request.payload);
        rollup.emitOutput(encodeNotice(advance.payload));
        return true;
    },
    inspect(request, rollup) {
        rollup.emitReport(request.payload);
    },
});
```

Handlers receive the request with its **raw, undecoded payload**. For an advance, call [`decodeAdvance`](/reference/decode-advance) to read the input fields, and build provable outputs with [`encodeNotice`](/reference/encode-notice) / [`encodeVoucher`](/reference/encode-voucher) before passing them to [`emitOutput`](/reference/emit-output).

## Returns

`Promise<never>` — the loop runs until `waitForInput` fails (inputs exhausted on the mock, or device closed), and the promise rejects with that error. It never resolves.

## Parameters

### handlers

- Type: `RunHandlers`

An object with an optional handler per request type. Both receive `(request, rollup)` and may be synchronous or `async`:

```ts twoslash
import type { AdvanceRequest, InspectRequest, Rollup } from '@tuler/node-libcmt';
// ---cut---
type RunHandlers = {
    advance?: (
        request: AdvanceRequest,
        rollup: Rollup,
    ) => boolean | void | Promise<boolean | void>;
    inspect?: (
        request: InspectRequest,
        rollup: Rollup,
    ) => boolean | void | Promise<boolean | void>;
};
```

The handler's result decides the request's fate:

| Handler outcome | Effect |
| --- | --- |
| returns `true` or `undefined` | request **accepted** |
| returns `false` | request **rejected** |
| throws | error emitted as a [report](/reference/emit-report), request **rejected** |
| no handler for the request type | request **rejected** |

See [Handling Requests](/guide/handling-requests) for the semantics of accept/reject.
